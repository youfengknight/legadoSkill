# SKILL 高频技巧速记

这份文档收集原本适合放在 `SKILL.md`、但又值得长期保留的高频实战技巧。

适用场景：

- 快速创建书源时查速记
- 修复已有书源时查常见坑
- 不想翻长教程，只想确认关键规则时查阅

## 1. 源码优先原则

模拟调试结果不稳定时，以 Legado 官方源码为准。

关键位置：

| 功能 | 源码位置 |
| --- | --- |
| CSS 选择器 | `legado/app/src/main/java/io/legado/app/model/analyzeRule/` |
| 书源规则 | `legado/app/src/main/java/io/legado/app/data/entities/BookSource.kt` |
| 搜索规则 | `legado/app/src/main/java/io/legado/app/model/SearchBook.kt` |

使用原则：

- 模拟结果和实际行为冲突时，优先读源码
- 不要把模拟引擎行为当成最终标准
- 最终仍需在 Legado APP 中实测

## 2. 搜索接口发现技巧

创建书源时，搜索接口发现通常是第一步关键工作。

建议顺序：

1. 先看搜索表单和 input 字段名
2. 再看内联 JavaScript
3. 再看外部 JavaScript
4. 再测试常见搜索接口格式
5. 多次失败后再回看案例

常见搜索字段名：

```text
searchkey, q, wd, query, search, key, keyword, value, s, word
```

优先尝试顺序：

```text
searchkey -> q -> wd -> keyword -> search
```

补充建议：

- 优先用长度 3 个字及以上的书名搜索
- 避免“我的”“系统”这类噪声关键词
- 搜索无结果时，先检查字段名、编码、请求方式，再怀疑选择器

## 3. 编码判断技巧

编码问题是书源调试里的高频坑点。

建议顺序：

1. 先看 HTML 中的 `meta charset`
2. 再看响应头中的 `Content-Type`
3. 看搜索结果是否出现乱码或完全为空
4. 对笔趣阁类站点优先怀疑 `GBK`

经验规则：

- 中文搜索出现乱码，优先检查编码
- 中文搜索无结果，也要怀疑编码而不是只怀疑接口
- 小说站点常见 `GBK`
- 不要只看页面展示正常就默认接口一定是 `UTF-8`

## 4. 发现规则技巧

`exploreUrl` 必须和站点导航里的真实分类链接格式保持一致，不能凭经验臆造。

编写步骤：

1. 找到首页导航中的真实分类链接
2. 分析 URL 规律
3. 只把页码位置替换成 `{{page}}`
4. 不要顺手改写路径层级、分隔符、后缀

注意事项：

- 导航链接怎么写，`exploreUrl` 就按那个格式来
- 不能因为别的站常见某种格式，就套到当前站
- 如果分类规则不稳定，宁可暂不交付 `exploreUrl`

## 5. 正则替换速记

Legado 书源 JSON 里的正则，反斜杠不需要额外转义。

错误示例：

```text
##\\[|\\]##
##\$$|\$$##
```

正确示例：

```text
##\[|\]##
```

常见替换格式：

| 格式 | 示例 | 说明 |
| --- | --- | --- |
| `##正则` | `##文字` | 删除匹配内容 |
| `##正则##替换` | `##文字##替换` | 替换匹配内容，返回整个字符串 |
| `##正则##替换###` | `##文字##替换###` | 只返回替换后的内容，匹配失败返回空 |

理解要点：

- `###` 结尾表示只取替换结果
- 不带 `###` 时，返回的是整个处理后的字符串
- 导入失败时，要优先排查是否多写了反斜杠

## 6. 规则兼容性速记

以下写法应避免：

- `prevContentUrl`
- `:contains()`
- `:first-child`
- `:last-child`

替代原则：

- 正文分页只考虑 `nextContentUrl`
- 文本筛选优先使用 `text.文本`
- 节点定位优先使用索引，如 `.0`、`.-1`

## 7. 字段与输出细节

### `concurrentRate`

推荐写法：

```json
"concurrentRate": ""
```

说明：

- 空字符串表示不限制并发率
- 不要默认写 `"0"`，容易带来额外风险

### 文件保存位置

推荐保存到：

```text
temp/书源名称/
```

不要直接放到项目根目录。

### JavaScript 注释

复杂 JavaScript 除了能跑，还应写清楚用途注释，至少说明：

- 这段 JS 在处理什么问题
- 输入来源是什么
- 输出结果会写到哪个字段

## 8. 失败排查优先级

书源不工作时，建议按这个顺序排查：

1. 输入 URL 是否真实有效
2. 请求方式是否正确
3. 编码是否正确
4. 页面结构是否拿对
5. 选择器是否匹配
6. 正则替换是否写错
7. 是否存在登录、验证码、Cloudflare、加密参数

原则：

- 先排除基础输入问题
- 再排选择器和规则问题
- 最后再进入复杂反爬处理
