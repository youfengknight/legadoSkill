# SKILL.md 重构方案

生成时间：2026-03-21

## 重构目标

`SKILL.md` 现在最大的问题不是内容少，而是主入口承担了太多“教程、架构、脚本说明、增强能力说明”，导致主流程不够清晰。重构目标应该是：

1. 让模型先快速判断“这是不是书源任务”。
2. 让模型明确知道“该按什么流程做”。
3. 让模型明确知道“哪些是硬约束，哪些只是可选增强”。
4. 让模型在缺信息时，知道该补什么，而不是直接开做。
5. 把深度资料下沉到 `references/`，让 `SKILL.md` 只保留主控信息。

## 建议结构

建议把 `SKILL.md` 控制在 250-400 行左右，结构改成下面这样：

1. YAML 头
2. 技能定位
3. 适用范围与非适用范围
4. 核心工作模式
5. 标准主流程
6. 输入契约
7. 输出契约
8. 强约束
9. 可选增强能力
10. 失败与降级策略
11. 参考文档导航

## 每一部分该写什么

### 1. YAML 头

只保留一份，不能重复。现在这个是必须先清理的。

建议：

```yaml
---
name: legado-book-source-tamer
description: 用于创建、修复、调试、讲解 Legado 阅读书源。涉及书源规则、站点分析、搜索/详情/目录/正文规则时必须使用。
---
```

### 2. 技能定位

这里不要再堆太多背景，只回答三个问题：

- 这个 skill 是干什么的
- 什么时候必须用
- 最终目标是什么

建议写法：

- 创建 Legado 书源
- 修复已有书源
- 分析站点结构与规则
- 解释 Legado 规则语法
- 最终产物是可导入 Legado 的 JSON 或明确的修复方案

### 3. 适用范围

建议列清楚触发词和任务类型。

适用：

- 新建小说/漫画/听书/订阅源
- 修复搜索、详情、目录、正文问题
- 分析 API、HTML、Ajax、加密参数
- 解释 CSS/JSONPath/XPath/JS 写法

非适用：

- 单纯小说推荐
- Legado 下载/安装
- 非书源技术问题

这一段的作用是防止 skill 误触发或漏触发。

### 4. 工作模式

建议从现在的“知识对话 / 完整生成”扩成 3 模式，更实用：

#### 模式 1：知识问答

- 解释规则
- 举例
- 不创建书源

#### 模式 2：书源修复

- 用户给现成 JSON 或报错现象
- 分析问题
- 输出修复后的规则或 JSON

#### 模式 3：新建书源

- 从网站或接口出发
- 分析并生成新源

这样比现在更贴合真实使用场景。

### 5. 标准主流程

这是 `SKILL.md` 最核心的部分，建议写得短而硬，不要掺杂太多扩展能力。

建议主流程固定成 5 步：

#### 第一步：任务分类

判断：

- 问答 / 修复 / 新建
- 搜索源 / 完整源
- HTML / JSON API / 动态加载
- PC / 移动端优先

#### 第二步：信息收集

收集：

- 站点 URL
- 搜索页
- 详情页
- 目录页
- 正文页
- 请求方式
- 编码
- 是否登录/验证码/Cloudflare

#### 第三步：规则分析

输出对应规则草案：

- `searchUrl`
- `ruleSearch`
- `ruleBookInfo`
- `ruleToc`
- `ruleContent`
- `exploreUrl`
- `ruleExplore`

#### 第四步：规则产出

整理成完整 JSON。

#### 第五步：结果校验

检查：

- JSON 结构
- 必填字段
- 选择器合法性
- 风险说明
- 是否需要用户实测

重点是：这里不要把“多智能体、上传、批量抓取”写进主流程。

### 6. 输入契约

这个是现在最缺的，建议明确区分“最少输入”和“推荐输入”。

#### 最少输入

- 目标站点 URL
- 任务类型：新建还是修复
- 目标功能范围：只搜索 / 搜索+详情 / 完整源

#### 推荐输入

- 搜索关键词样例
- 书籍详情页 URL
- 目录页 URL
- 正文页 URL
- 现成书源 JSON
- 报错截图或现象
- 是否优先手机站或 PC 站

这一段写好以后，skill 就不容易在信息不足时乱推断。

### 7. 输出契约

建议强制规定输出格式，避免每次产出不一致。

建议固定包含：

- 最终 JSON
- 保存路径
- 本次已验证的页面类型
- 尚未验证的页面类型
- 可能存在的风险
- 需要用户在 Legado APP 中实测的项目

比如：

```md
输出必须包含：
1. 完整书源 JSON
2. 保存路径，例如 temp/站点名/站点名.json
3. 已完成功能说明
4. 未完成或未验证项
5. 风险说明
6. 建议用户在 Legado 中实测的步骤
```

### 8. 强约束

这一段保留硬规则，不要混入建议项。

建议只留真正高价值的约束：

- 不允许编造站点规则
- 必须基于真实页面结构或用户提供内容分析
- 不要把可选能力当必选流程
- 只要有下一页按钮，就要考虑 `nextContentUrl`
- 禁止使用不兼容写法，如 `:contains()`、`prevContentUrl`
- 复杂 JS 必须附用途说明
- 最终结果必须提示用户在 Legado 中实测

这部分要短，不然模型抓不住重点。

### 9. 可选增强能力

现在这块应该从主流程里挪出来。

建议单独列为“可选增强能力”：

- 批量抓取
- 多智能体并行分析
- 上传书源获取直链
- 调试器验证
- 特殊站点反爬处理

并明确一句：

“仅当环境支持、任务复杂且确有必要时使用；不能作为默认必走流程。”

这样就能解决现在 skill 里最容易带偏模型的问题。

### 10. 失败与降级策略

建议这是重构里第二重要的部分。

建议写成明确规则：

- 搜索可做但详情未确认：允许交付搜索源
- 详情和目录可做但正文分页不稳：允许交付部分可用源，并标注风险
- 站点需要登录、验证码、Cloudflare 且无法验证：中止完整交付，改为给分析结论和待验证点
- 用户提供信息不足：先索要最小补充信息，不直接臆造

这会比现在那种“容错架构图”更实用。

### 11. 参考文档导航

这里只保留导航，不复制内容。

建议按用途分组：

- 入门：`QUICKSTART.md`、`FAQ.md`
- 主流程：`references/workflow-guide.md`
- 语法：CSS、Regex、JSON、JS
- 特殊场景：API、POST、动态加载、验证码、WebView
- 案例：`examples/README.md`

并确保所有链接都存在。

## 建议删掉或下沉的内容

这些内容不建议继续放在主 `SKILL.md` 里：

- 大段多智能体架构图
- `Task(...)` 调度示例
- 过细的脚本文件清单
- 太长的教程摘录
- 可选上传流程
- 复杂案例长篇展开

这些应该下沉到：

- `references/`
- `scripts/`
- `examples/`

`SKILL.md` 只负责“调度”。

## 重构后的风格建议

建议整体风格从“百科全书型”改成“作战手册型”：

- 少解释背景
- 多写判断标准
- 多写输入输出
- 多写禁止项和降级项
- 所有扩展内容尽量跳转到 `references/`

## 建议优先级

先做：

1. 清理 YAML 头重复定义
2. 修正错误链接
3. 重写主流程
4. 增加输入契约
5. 增加输出契约
6. 增加失败降级策略
7. 把多智能体/上传改为可选增强

后做：

1. 优化案例索引
2. 新增决策树文档
3. 新增环境边界说明
4. 新增调试检查清单

## 一句话方案

把 `SKILL.md` 从“知识汇总文档”重构成“主控调度文档”：主流程更短，约束更硬，输入输出更清楚，增强能力降级为可选，深度资料全部下沉到 `references/`。
